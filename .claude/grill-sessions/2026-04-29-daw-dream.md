# Grill Session: In-browser DAW dream & overarching plan

**Date:** 2026-04-29
**Status:** completed

## Decision Branches
- [x] 1. Dream scope (what's in / out) → **E** (Studio + ecosystem + collab)
- [x] 1a. Collab depth → **B** (edit-time + shared transport, no audio/MIDI streaming)
- [x] 2. Audio graph model → **C** (fixed strip + container plugins as escape hatch)
- [x] 3. Plugin architecture → **B** (WASM DSP + JS UI; first-party as static Rust on same trait)
- [x] 4. Timing & transport → **C** dual timeline (musical time PPQ + sample anchors; MIDI=musical, audio=sample default w/ opt-in warp; tempo map)
- [x] 5. Content model → **C** strict typed tracks (Audio/MIDI/Bus); MIDI tracks hold piano-roll OR step-seq clips. Session-view = stretch goal, clips first-class.
- [x] 6. State & persistence → **Yjs** (project = Y.Doc; awareness for cursors/transport; y-indexeddb for offline; audio engine reads snapshots not CRDT)
- [x] 7. Threading topology → **C** hybrid (structural edits via postMessage; param/note/transport via SAB ring). COOP/COEP required.
- [x] 8. UI shell & PIP → **D** Document PIP for transport (always-on-top) + `window.open` popups for big-panel side-by-side. One Y.Doc shared via BroadcastChannel; audio engine stays in root window.
- [x] 9. MIDI I/O → **C** input + CC learn + shareable controller-map presets. MIDI out & clock-master = future, not architectural blocker.
- [x] 10. Voice management → **B** per-instrument pool, user-resizable (1–128). Future: optional global budget mode.
- [x] 11. Audio assets & recording → **C** OPFS local cache + cloud object store; content-addressed; Y.Doc holds hash+url+metadata only.
- [x] 12. Build / distribution → **D** local-first first, hosted-collab later. Day-one architecture is collab-ready (Yjs shapes, awareness, content-addressed assets), no servers run until needed.
- [x] 13. Phase slicing philosophy → **D** capability-driven phases (each = "user can do X"). Phase 1 = existing POC. Each phase spans layers; rework is informed by previous phase's lessons.

## Q&A

### Q1 of ~16: Dream scope
**Question:** How wide is the dream / what's the broadest tier the architecture must accommodate?
**Options:** A. Sequencer-plus / B. +Audio clips / C. +Live recording & MIDI in / D. +Plugin SDK / E. +Multi-user collab
**Answer:** E
**Recommendation:** D — collab warps every data-model decision around CRDTs.
**Outcome:** E. Data model must be CRDT-compatible from day one. Pushes "edits are events" discipline early.

### Q2 of ~17: Collab depth
**Question:** How deep does multi-user go (edit-only / +shared transport / +MIDI jam / +audio streaming)?
**Options:** A. Edit-only / B. +Shared transport / C. +MIDI jam / D. +Audio streaming
**Answer:** B
**Recommendation:** B — Figma-style edits + magical shared transport without WebRTC complexity.
**Outcome:** B. CRDT covers project state. Shared transport via clock-sync messages over a signaling channel. Audio rendered locally per client. Door left open for (C/D) as opt-in later if event bus is clean.

### Q3 of ~17: Audio graph model
**Question:** Fixed channel strip vs free-form node graph?
**Options:** A. Fixed strip / B. Free-form node graph / C. Fixed strip + container-plugin escape hatch
**Answer:** C
**Recommendation:** C — sane UI + predictable CPU + simple CRDT shape, with graph flexibility delegated to plugins.
**Outcome:** C. Project shape: ordered list of tracks, each = `{clips, instrument slot, insert FX list, send list, gain/pan/mute}`. Buses are tracks too. Container plugins can host mini-graphs internally; host doesn't expose graph editing.

### Q4 of ~17: Plugin ABI
**Question:** What's the binary contract for plugins (synths/FX/samplers/containers)?
**Options:** A. WASM-only, host-rendered UI / B. WASM DSP + JS UI / C. Static Rust trait only / D. AudioWorkletProcessor plugins
**Answer:** B
**Recommendation:** B — same plugin trait used two ways: first-party static-Rust plugins for the fast path, third-party WASM+JS plugins for the SDK.
**Outcome:** B. Plugin = (1) DSP module exposing `init/process/set_param/descriptors`, (2) optional JS UI bundle mounted in a host-provided container. First-party plugins compile against the same trait directly into the engine; third-party plugins ship as WASM+JS pairs loaded by URL.

### Q5 of ~17: CRDT framework
**Question:** Which CRDT for project state + collab transport?
**Options:** A. Yjs / B. Automerge / C. Loro / D. Custom op-log
**Answer:** A
**Recommendation:** A — best tooling, awareness API solves cursors + shared transport, mature ecosystem.
**Outcome:** A (Yjs). Project = Y.Doc. Awareness API carries cursor/selection/transport position. y-websocket for sync, y-indexeddb (or OPFS adapter) for offline. Audio engine consumes immutable snapshots, never the CRDT directly.

### Q6 of ~17: State → audio thread
**Question:** How do project edits reach the AudioWorklet?
**Options:** A. postMessage snapshot-on-change / B. SAB ring buffer of commands / C. Hybrid (postMessage + SAB) / D. No worklet
**Answer:** C
**Recommendation:** C — frequent tiny edits via SAB (sample-accurate), rare bulky edits via postMessage.
**Outcome:** C. Two queues into engine: slow "rebuild" via postMessage (add track, swap plugin, load sample), fast "events" via SAB ring (note on/off, set_param, transport). COOP/COEP headers required → constrains plugin UI hosting (iframes need crossOriginIsolated).

### Q7 of ~17: Timing model
**Question:** How is musical/audio time represented internally?
**Options:** A. Sample-time only / B. Bars/beats only / C. Dual timeline (musical PPQ + sample anchor + tempo map) / D. Warp-everything
**Answer:** C
**Recommendation:** C — only model that handles tempo automation, time-sig changes, mixed MIDI+audio clips, and latency-compensated shared transport.
**Outcome:** C. Internal time = PPQ ticks (recommend 960). Tempo map = piecewise-linear function ticks↔samples. MIDI clips anchored to ticks. Audio clips default to sample anchor (natural rate), opt-in warp re-anchors to ticks. Shared transport syncs at ticks not samples.

### Q8 of ~17: Content model
**Question:** Strict typed tracks vs polymorphic tracks; does step-seq become a clip type?
**Options:** A. Strict typed (Audio/MIDI) / B. Polymorphic / C. Strict + step-seq as MIDI clip type / D. Add session-view launcher
**Answer:** C
**Recommendation:** C — clean mental model, clips are first-class so D becomes addable later.
**Outcome:** C. Tracks are `Audio | MIDI | Bus`. MIDI clips have variants `PianoRoll | StepSeq`. Audio clips have variants `Sample(WarpMode)`. Automation lanes are per-track per-parameter. Session-view = future, but design clips as standalone referencable objects (don't bake timeline position into the clip identity).

### Q9 of ~17: Voice management
**Question:** How are voices allocated/stolen across many polyphonic tracks?
**Options:** A. Per-instrument fixed pool / B. Per-instrument user-sized pool / C. Global host-managed budget / D. Unbounded
**Answer:** B
**Recommendation:** B — where most DAWs land; A is a footgun, C complicates plugin ABI for marginal gain.
**Outcome:** B. Each instrument plugin instance owns a pool sized 1–128 (default 16). Plugin handles its own stealing (oldest/quietest). Future knob: opt-in global budget if real projects show cross-track starvation.

### Q10 of ~17: Multi-window / PIP
**Question:** Chrome Document PIP usage for multi-pane control?
**Options:** A. PIP for transport only / B. PIP as panel detacher / C. window.open multi-window / D. Hybrid PIP+window.open
**Answer:** D
**Recommendation:** D — burn the single PIP slot on transport (always-on-top key feature), use plain popups for everything else.
**Outcome:** D. Architecture: root window hosts the audio engine + Y.Doc + WS sync. Satellite windows (PIP transport, popup mixer, popup piano-roll, popup plugin UIs) are render-only, share state via BroadcastChannel + a snapshot of the Y.Doc. Same-origin requirement for COOP/COEP.

### Q11 of ~17: Audio assets storage
**Question:** Where do audio clips/recordings live & how does sharing work?
**Options:** A. OPFS only / B. IndexedDB blobs / C. OPFS cache + cloud object store / D. Server-only streaming
**Answer:** C
**Recommendation:** C — collab needs shared fetchable assets; offline needs local cache. Content-addressed is the right glue.
**Outcome:** C. Assets identified by `sha256` hash. OPFS is local cache. Canonical store = cloud bucket (S3/R2). Y.Doc holds `{hash, url, channels, sampleRate, frames}` per asset, never PCM. Recording: stream to OPFS first, hash on close, background-upload. Project export = zip of Y.Doc snapshot + referenced assets.

### Q12 of ~17: MIDI I/O
**Question:** How deep on WebMIDI input, CC mapping, and external MIDI output?
**Options:** A. Input only, fixed routing / B. + CC learn / C. + Shareable controller-map presets / D. Full I/O incl. external clock
**Answer:** C
**Recommendation:** C — significant UX uplift for hardware users; D is a future project.
**Outcome:** C. WebMIDI input → armed track + CC-learn binds physical controls to any parameter. Per-device map presets stored separately from project. Output to external gear = future. MIDI clock master = future.

### Q13 of ~14: Distribution & first-user experience
**Question:** What's the minimum shipping shape and how does collab infra fit in?
**Options:** A. Self-hosted only / B. Hosted SaaS / C. Local-first + optional hosted / D. Local-first first, hosted later
**Answer:** D
**Recommendation:** D — matches the "build slow, test often" cadence; architecture is collab-ready from day one but no service runs until needed.
**Outcome:** D. Phase 1 = offline single-user. All data shapes designed Yjs-aware, content-addressed, awareness-friendly. Hosted sync server + asset bucket = a later flip-the-switch phase, not a rewrite.

### Q14 of ~14: Phase slicing philosophy
**Question:** How do we cut the dream into phases?
**Options:** A. Vertical thin-slice / B. Horizontal layered / C. Hybrid (one vertical, then horizontal expansion) / D. Capability-driven (each phase = a verb the user can do)
**Answer:** D
**Recommendation:** D — every phase is dogfoodable, every phase spans layers, rework is informed by lessons.
**Outcome:** D. Phases are headlined by "user can do X." Phase 0 = existing POC (already shipped). Subsequent phases each deepen across the stack.
